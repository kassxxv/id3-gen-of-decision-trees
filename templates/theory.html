ID3 Algorithm Quinlan, 1986

ID3 — Iterative Dichotomiser 3 — builds a decision tree by repeatedly splitting a dataset on the feature that reduces class uncertainty the most. It works top-down, greedily, with no backtracking.

Each internal node in the resulting tree tests one feature. Each branch is a possible value of that feature. Each leaf is a class label — assigned by majority vote over whatever training examples ended up there.

The algorithm only handles categorical features. Continuous values need to be discretised before training. This is a real limitation — C4.5 (Quinlan, 1993) extended ID3 to handle numeric splits and missing values.

The name "Iterative Dichotomiser" is slightly misleading — the algorithm is not iterative in the usual sense. It's recursive. The name refers to the fact that Quinlan developed it through several iterations of an earlier algorithm.
Entropy H(S)

Entropy measures how mixed up a set of examples is. If every example has the same label, entropy is 0 — no disorder, nothing to learn. If labels are split perfectly evenly, entropy is at its maximum.

H(S) = −∑ p𝑖 · log₂(p𝑖) p𝑖 = fraction of examples in S that belong to class i

For a binary problem, maximum entropy is 1.0 (50/50 split). For c classes, the ceiling is log₂(c). Entropy is always ≥ 0 and equals 0 only when the set is pure.

example — 9 positive, 5 negative: p₊ = 9/14 ≈ 0.643 p₋ = 5/14 ≈ 0.357 H(S) = −(0.643 · log₂ 0.643) − (0.357 · log₂ 0.357) ≈ 0.940
Information Gain IG(S, A)

Information Gain answers: how much does splitting on feature A reduce the entropy of set S? ID3 picks whichever feature maximises this at each node.

IG(S, A) = H(S) − ∑v (|S𝑣| / |S|) · H(S𝑣) S𝑣 = subset of S where A = v weight = proportion of examples that take value v

The weighted sum accounts for the fact that a split producing one large pure child and one tiny impure child is still mostly good. Larger subsets count more.

same example — feature "Wind": S = {9+, 5−} H(S) = 0.940 Wind=Weak → S𝑣 = {6+, 2−} H = 0.811 weight = 8/14 Wind=Strong → S𝑣 = {3+, 3−} H = 1.000 weight = 6/14 IG(S, Wind) = 0.940 − (8/14)·0.811 − (6/14)·1.000 ≈ 0.048
High-arity features (like a customer ID column) get artificially inflated IG because they split the data into many tiny pure subsets. C4.5 corrects this with Gain Ratio, which divides IG by the entropy of the feature's own value distribution.
The Algorithm Recursive
function ID3(S, features, target): if all labels in S are identical: return Leaf(that label) if features is empty: return Leaf(majority label in S) A = argmaxf IG(S, f) ← greedy choice node = Node(feature = A) for each value v in A: S𝑣 = { x ∈ S | x[A] = v } if S𝑣 is empty: node.add_child(v, Leaf(majority label in S)) else: node.add_child(v, ID3(S𝑣, features − {A}, target)) return node
  1. 01
    Pure node All examples agree — return a leaf immediately. No further splitting is possible or useful.
  2. 02
    No features left Examples still disagree but there's nothing left to split on. Return a leaf with the majority class. This is the fallback for contradictory data.
  3. 03
    Greedy split Pick the feature with the highest IG. No backtracking — this decision is permanent. The tree is built in one pass.
  4. 04
    Empty branch A feature value that never appears in S produces an empty subset. Use the parent's majority class as a default leaf.

Known Limitations ID3 only

ID3 was the starting point, not the final word. Quinlan himself addressed most of these in C4.5 seven years later.

ProblemWhat happensC4.5 fix
Categorical only No numeric features. Everything must be discretised beforehand. Threshold splits — finds the cut point that maximises gain
High-arity bias A feature with 100 unique values will almost always win IG, even if it's useless (e.g. a row ID). Gain Ratio divides IG by the feature's own split entropy
No missing values A missing value produces an empty branch. ID3 has no strategy for it. Fractional examples — split each missing example across branches by proportion
Overfitting The tree grows until every leaf is pure or features run out. On noisy data it memorises noise. Post-pruning — collapse subtrees where pruning reduces estimated error
Overfitting is the most practically painful issue. A tree that scores 100% on training data and 60% on test data is not a good model — it has memorised the training set rather than learned a general pattern. This implementation does not prune.
